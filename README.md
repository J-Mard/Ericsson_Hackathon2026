# Ericsson Hackathon 2026 — Subsea Welding Ops

A proof of concept we built in one sitting for the 6G track. The scenario: a
small Pacific-coast fleet of offshore rigs, each running an autonomous
underwater welding drone that's tethered by fibre to a 6G edge buoy on the
surface. The buoy is where the "thinking" happens, and the whole demo is
about showing how a sub-millisecond AI-native edge can keep those drones
safe when the weather goes sideways — and hand control back to a human the
moment the drone sees something it can't classify.

The app in this repo is the buoy's operator view.

## Run it

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
streamlit run frontend/app.py
```

Narration is generated by a local Qwen 2.5 3B model over Ollama. It's optional
— if Ollama isn't running, the chat falls back to short templates so the demo
doesn't stall. To enable the real model:

```bash
ollama pull qwen2.5:3b
ollama serve   # usually already running in the background
```

## What you see on screen

Top bar shows the clock, a simulated 6G URLLC latency read-out (100–500 µs,
i.e. well under the 1 ms mark that defines the 6G URLLC target), a session
event counter, and the Trigger Disaster button.

Below that, a **fleet map** plots both rigs at their real coordinates on the
California Pacific coast. Each pin is coloured by the live classifier state
(green = NORMAL, yellow = CAUTION, red = ABORT), and when a rig is in
takeover it gets an extra red ring around it. The dotted line between the
two pins is pure visual shorthand for the 6G edge mesh — it isn't modelling
anything, it just sells the "one edge, multiple remote assets" framing.

Below the map, two rig cards side by side:

- **Rig A — San Francisco bay**, fed by `open-meteo-37.79N122.46W18m.csv`
  (displayed in-app as `sf_underwater.csv`).
- **Rig B — Los Angeles basin**, fed by `LA_underwater_data.csv`.

Each card has its own reliability, wave height, lambda readout, live charts,
and a scrolling chat log between that rig's drone and its buoy.

Sidebar controls:

- **Tick interval** (ms) — how fast new rows are pulled.
- **Stress test** slider — multiplies wave height / current on top of whatever
  the CSV is serving. A clean way to push a calm day into CAUTION without
  going full storm.
- **LLM narration** toggle.

The **Trigger Disaster** button switches each rig onto its own hand-crafted
storm CSV (`storm_sf.csv` for Rig A, `storm_la.csv` for Rig B) for the next
30 ticks. The two files encode the same storm system as seen from each rig:
SF takes the hit first and harder, LA lags roughly an hour behind with
slightly lower peaks because of the partial Channel-Islands shelter. Both
feeds oscillate — three bands of bad weather with brief calm windows
between them — so the dashboards genuinely walk back and forth through
NORMAL → CAUTION → ABORT → CAUTION → NORMAL → CAUTION → ABORT → ... over
the 30 ticks, rather than a single clean build-peak-subside curve. The
data source badge on each rig card flips to `storm_sf.csv` / `storm_la.csv`
while the storm is active so the audience can see the swap.

## Data

Everything on screen comes from a CSV the jury can open:

| File | Use |
| --- | --- |
| `open-meteo-37.79N122.46W18m.csv` | Rig A steady-state (real Open-Meteo marine data) |
| `LA_underwater_data.csv` | Rig B steady-state (real Open-Meteo marine data, independent timeline) |
| `storm_sf.csv` | 30-hour oscillating storm arc for Rig A, hand-crafted |
| `storm_la.csv` | Same storm system from Rig B's viewpoint — phase-lagged, muted peaks |

No live API calls during the demo. It's deliberate: the stage wifi is not our
friend, and replaying CSVs is reproducible.

`LA_underwater_data.csv` also carries a `strange_geometry` column. A `1` in
that column at a given tick is our way of saying "the drone's onboard scan
found something it can't interpret" — see the human-in-the-loop section below.

## The math and ML

All of the decision logic lives in `core/ml.py`. Two pieces.

### Reliability curve

Standard exponential reliability model:

```
R(t) = exp(-λ · t)
```

We evaluate R at a fixed horizon so the KPI card gives a comparable number
between ticks. `λ` is a linear combination of the live telemetry:

```
λ = 0.05 · H_s + 0.03 · U_c
```

Where `H_s` is significant wave height in metres and `U_c` is ocean current
in m/s. The coefficients are hand-tuned so that calm conditions give a slow
decay and storm conditions collapse R fast enough to force an ABORT.

A few reference values:

| Scenario | H_s | U_c | λ | R at t = 10 s |
| --- | --- | --- | --- | --- |
| Calm | 0.5 | 0.3 | 0.034 | 0.71 |
| Building sea | 1.8 | 0.7 | 0.111 | 0.33 |
| Storm | 4.5 | 1.8 | 0.279 | 0.06 |

### Decision tree

A shallow `DecisionTreeClassifier(max_depth=5)` maps
`(wave_height, current_velocity, wave_period)` to one of
`NORMAL / CAUTION / ABORT`. We train it once per process on 3000 synthetic
rows labelled with transparent rules:

```python
if wave >= 3.0 or current >= 1.0:
    label = "ABORT"
elif wave >= 1.8 or current >= 0.6 or period < 5.0:
    label = "CAUTION"
else:
    label = "NORMAL"
```

Those thresholds come straight from the brief (`wave > 3 m` or
`current > 1 m/s` ⇒ emergency retreat). The tree learns them back, which is
the point — the model is easy to explain, inference is sub-millisecond, and
the predictions track an expert heuristic so a reviewer can sanity-check them
at a glance.

Inference happens once per tick per rig.

## Human-in-the-loop

When the `strange_geometry` flag trips on Rig B's CSV, we assume the drone
saw geometry it can't classify — a defect, silt, an obstruction, whatever.
The app latches that rig into a takeover state: a red banner appears at the
top, the rig card gets a `HIL · takeover` badge, and the chat thread switches
to a short handover exchange. It stays latched until the operator clicks
**Acknowledge takeover — Rig X**, at which point the agents resume routine
narration.

This is the pitch for a low-latency edge: routine telemetry is handled on the
buoy, but the moment something ambiguous shows up, the buoy hands control up
to a human over a 6G URLLC slice without a full round-trip to a distant
cloud region. The KPI card shows latencies in the 100–500 µs range, which
is the 3GPP 6G URLLC working target (sub-millisecond, one to two orders of
magnitude tighter than 5G URLLC's ~1–10 ms).

## Architecture

```
 sf_underwater.csv  ─┐
 LA_underwater.csv  ─┤
 storm_sf.csv       ─┼─► core/data.py ─► per-rig Tick
 storm_la.csv       ─┘        ▲
                       disaster toggle, stress slider
                               │
                               ▼
                       core/ml.py   (tree + λ, R(t))
                               │
                   state change or HIL?
                               │
                               ▼
                       core/llm.py  (Qwen 2.5 3B, template fallback)
                               │
                               ▼
                     frontend/app.py  (Streamlit fragment)
                         ├─ fleet map (plotly scattergeo)
                         ├─ per-rig KPI cards + charts
                         └─ chat panels
```

The UI uses `st.fragment(run_every=...)` so only the live panels re-render on
each tick — the page itself doesn't reload. The fleet map is re-built inside
the same fragment, so its pin colours stay in sync with the KPI cards.

## Things we intentionally left out

- No live weather API. Deterministic replay only.
- No parallel LLM calls. We narrate sequentially and only on state changes,
  which keeps the M3's thermal situation boring.
- No persistence. Everything lives in `st.session_state`; hitting Reset in
  the sidebar wipes the history and counters.
